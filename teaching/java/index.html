Modification de Piece.java Modification du fichier Piece.java. On commence par ajouter les attributs permettant de gérer les objets d’une pièce avec un tableau (n’hésitez pas à chercher sur internet si vous ne savez pas comment ça marche). On utilise un tableau simple en Java mais on verra plus tard dans le cours des listes plus pratiques. On ajoute un nouveau constructeur (il est possible d'avoir plusieurs constructeurs avec paramètres différents) qui permet d'initialiser les objets présents dans la pièce au départ. On ajoute les méthodes que l’on veut. J’en propose certaines qui me viennent à l’esprit mais vous pouvez ajouter ce que vous voulez. J’avais proposé une méthode compliquée à certains en TP pour gérer le tableau mais je pense que je vais proposer plus simple. Par exemple, pour ajouter un élément, je construis un nouveau tableau plus grand qui va remplacer le tableau d’objets de la salle. Pour supprimer, je mets null à la place de l’objet. Puis on ignorera à chaque fois les cases qui contiennent null. Plus tard, on découvrira des listes plus pratiques pour gérer ça comme ArrayList. Je vous propose ensuite de faire quelques tests dans Jeu.java pour ajouter des objets dans certaines pièces (en modifiant la méthode creerPieces) puis afficher automatiquement la liste des objets (en modifiant la méthode deplacerVersAutrePiece) quand on entre dans une salle. Vous pouvez ensuite lancer le programme pour voir si ça fonctionne.
Equals & Hashcode pour ObjetZork.java Il est parfois utile de redéfinir la méthode equals qui permet de dire si deux objets sont égaux pour la personnaliser et choisir nous-même ce que signifie être égal pour deux objets. Par exemple : Tester l’égalité peut être coûteux et inefficace dans certains cas où on doit faire énormément de comparaisons. Pour cette raison, Java stocke un code dans les objets qui s’appelle hashcode. Le principe est que si deux objets sont censés être égaux alors ils doivent avoir le même hashcode (mais la réciproque n’est pas forcément vraie, on a pas besoin d’unicité). Pour redéfinir le hashcode, doit respecter le “contrat” qui donne des indications sur ce qu’on doit respecter pour la définition. Pour des raisons techniques, dans la documentation Java, il est souvent choisi de multiplier le hashcode des chaînes de caractère par 31 (je rappelle que les chaînes de caractère sont des objets). Joueur.java C’est presque la même chose que la classe Piece.java: les deux sont des objets qui peuvent contenir des ObjetZork. Il y a donc à peu près les mêmes fonctions. La seule différence est le sens : on “ajoute” ou “supprime” un objet d’une salle mais un joueur “porte” ou “jette” un objet. Grâce à l’héritage de classe, il est possible de dire que les deux classes sont filles d’une autre classe plus générale permettant le transport d’ObjetZork.
